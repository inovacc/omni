# ADR-0003: Internalize protocompile

## Status

**Proposed** — 2026-02-18

## Context

`github.com/bufbuild/protocompile` v0.14.1 is used by omni's `buf build`, `buf format`, and `buf lint` commands. Currently it's an external dependency. This ADR evaluates whether to internalize it.

## Analysis

### Summary

| Field | Value |
|-------|-------|
| Module | `github.com/bufbuild/protocompile` |
| Version | v0.14.1 |
| Total Go files | ~322 (247 non-test + 75 test) |
| Packages | ~55-60 |
| Direct deps | 13 (6 prod, 7 test-only) |
| License | Apache 2.0 (compatible with BSD-3) |
| Upstream commit | `d02d3732d1132a0f770aa934503e5a93356883f9` |

### Packages Used by Omni

| Package | Used In | Purpose |
|---------|---------|---------|
| `protocompile` (root) | build.go | `Compiler`, `SourceResolver`, `WithStandardImports` |
| `protocompile/ast` | format.go, lint.go | `FileNode`, token walking, AST node types |
| `protocompile/parser` | format.go, lint.go | `Parse()` — source to AST |
| `protocompile/linker` | build.go | `Files` — compiled output |
| `protocompile/reporter` | build.go, format.go, lint.go | Error handling |

Only **5 of ~60 packages** are actually used.

### Dependencies Brought Along

| Dependency | Required | Notes |
|-----------|----------|-------|
| `google.golang.org/protobuf` | Yes | Already in omni's go.mod |
| `golang.org/x/sync` | Yes | Used by compiler |
| `github.com/rivo/uniseg` | No | Only experimental/dom |
| `github.com/tidwall/btree` | No | Only internal/interval |
| `golang.org/x/exp` | No | Only internal/interval |
| Test deps (testify, go-cmp, etc.) | No | Strippable |

### Risk Assessment

- **License**: Apache 2.0 — fully compatible with omni's BSD-3
- **Maintenance burden**: ~247 non-test Go files, but only ~46 in the stable API (ast, parser, linker, reporter, etc.)
- **Complexity**: Generated parser (`proto.y.go` from goyacc), generated enum code, arena allocator
- **No CGO**: Pure Go, fully cross-platform
- **Active development**: Experimental next-gen compiler under heavy development — frequent upstream changes

## Decision

**Keep as external dependency.** Do not internalize.

### Rationale

1. **Minimal surface area**: Omni uses only 5 of ~60 packages. The dependency is well-scoped.
2. **High churn risk**: The protocompile repo is under active development with a next-gen compiler rewrite. Internalizing would freeze at v0.14.1 and miss improvements/bugfixes.
3. **Generated code**: The parser (`proto.y.go`) is generated by goyacc. Maintaining this without the build tooling adds friction.
4. **Transitive deps are small**: Only 2 new production deps (`golang.org/x/sync` + `google.golang.org/protobuf`, both already in go.mod).
5. **No forking needed**: Omni doesn't need to modify protocompile's behavior — it's used as-is.

### When to Reconsider

- If protocompile introduces breaking API changes that block omni upgrades
- If protocompile is abandoned or deprecated
- If omni needs custom parser/linker behavior that requires source modifications

## Alternatives Considered

1. **Full copy** (~322 files) — Too large, too much maintenance
2. **Subset copy** (~46 stable files + ~68 internal) — Still ~114 files with complex internal dependencies
3. **Fork & adapt** — No modifications needed currently

## Consequences

- protocompile stays in go.mod as a direct dependency (change from `// indirect`)
- Upgrade path: `go get github.com/bufbuild/protocompile@latest`
- ADR-0002 strategy (protocompile as dep instead of full buf internalization) is validated
