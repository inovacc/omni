# omni Hardening Audit

_Generated by omni-harden-audit workflow_

## Executive summary

This audit adversarially verified 51 findings across the omni codebase, spanning security (25), robustness (15), concurrency (3), and supply-chain (8) dimensions. The single most severe issue is a classic **tar-slip / zip-slip arbitrary-file-write primitive** in the archive extractor (`archive-01`, Critical), which is directly triggerable by extracting an untrusted archive — exactly omni's stated CI/CD threat model — and is compounded by symlink/hardlink escape variants and a missing decompression-bomb guard. The High-severity tier is dominated by a systemic breach of omni's foundational **NO-EXEC invariant**: at least eight production packages (`exec`, `forloop`, `task`, `terraform`, `git/hacks`, `gh/hacks`, `repo`, `buf/generate`, plus build-tagged `machineid_darwin`/`uname_windows`) spawn external processes — `sh`/`cmd`, `git`, `gh`, `protoc`, `terraform`, `ioreg` — undermining the "single static binary, no external processes" core value proposition, and two of them (`forloop`, `task`) additionally interpolate untrusted input into a system shell, yielding genuine command-injection sinks. The Medium and Low tiers are mostly defense-in-depth and robustness gaps: unbounded `io.ReadAll`/recursion/pagination DoS vectors against untrusted network and file input, a latent top-level panic that bypasses exit-code mapping and log finalization, world-readable secret-bearing files (logs, profiles, decrypt output), a non-cancellable goroutine deadlock in `rg`, weak/un-versioned KDF cost, and a missing `-trimpath` reproducibility gap. Overall posture: **the core utilities are sound, but the archive extractor and the accumulated exec escape-hatches are the two structural liabilities that must be closed before v1.0.** Most findings are self-inflicted or require local/operator preconditions; the archive extractor is the one that is remotely-supplied-data exploitable end-to-end.

#### [CRITICAL][security] tar-slip path traversal in extractTarArchive — archive entry names escape destination dir

- **File:** `internal/cli/archive/tar.go:173` (`extractTarArchive`)
- **CWE:** CWE-22
- **What/Why:** `header.Name` comes directly from the untrusted tar archive and is joined into the destination with `filepath.Join(destDir, name)` with no containment check. A crafted entry name such as `../../../../etc/cron.d/evil` resolves outside `destDir`, so extraction can create or overwrite arbitrary files (classic tar-slip / CVE-2018-1002200 class). `StripComponents` does not remove `..` segments. The resulting target feeds `MkdirAll`, `OpenFile` (TypeReg), `Symlink`, and `Link` — every write path is exploitable.
- **Evidence:**
  ```go
  target := filepath.Join(destDir, name)
  ...
  case tar.TypeReg:
      ...
      outFile, err := os.OpenFile(target, os.O_CREATE|os.O_WRONLY|os.O_TRUNC, os.FileMode(header.Mode))
  ```
- **Exploitability:** Directly exploitable with no privilege. Trigger is just `omni tar -xf evil.tar` / `omni unzip evil.zip`, matching omni's stated threat model (a tar/unzip replacement in CI/CD over untrusted artifacts). Two aggravating points: (1) the same unguarded join exists in `extractZipArchive` (`zip.go:121`) — any fix must patch both; (2) the `tar.TypeSymlink` case writes a symlink whose `Linkname` is fully attacker-controlled and unvalidated, enabling a secondary symlink-following write even if the immediate target is contained, so symlink/hardlink `Linkname` must also be validated.
- **Recommended fix:** After computing `target`, enforce containment: `cleanDest, _ := filepath.Abs(filepath.Clean(destDir)); cleanTarget := filepath.Clean(filepath.Join(cleanDest, name)); if cleanTarget != cleanDest && !strings.HasPrefix(cleanTarget, cleanDest+string(os.PathSeparator)) { return cmderr.Wrap(cmderr.ErrInvalidInput, "archive: entry escapes destination: "+header.Name) }`. Reject absolute entry names. Apply the guard before every `os.MkdirAll`/`OpenFile`/`Symlink`/`Link`.
- **Confidence:** high
- **Finding ID:** `archive-01`

#### [HIGH][security] zip-slip path traversal in extractZipArchive — zip entry names escape destination dir

- **File:** `internal/cli/archive/zip.go:121` (`extractZipArchive`)
- **CWE:** CWE-22
- **What/Why:** `f.Name` comes from the untrusted zip central directory and is joined with `filepath.Join(destDir, name)` with no containment check. A crafted entry name like `..\..\..\Windows\System32\x` (or POSIX `../../x`) escapes `destDir`, allowing arbitrary file creation/overwrite during unzip. `StripComponents` only drops leading components and does not strip `..` segments.
- **Evidence:**
  ```go
  name := f.Name
  ...
  target := filepath.Join(destDir, name)
  ...
  outFile, err := os.OpenFile(target, os.O_CREATE|os.O_WRONLY|os.O_TRUNC, f.Mode())
  ```
- **Exploitability:** Realistic. `filepath.Join` applies `filepath.Clean`, which collapses `../` segments, so an entry like `../../../some/file` escapes `destDir`, enabling arbitrary file creation/overwrite via `os.MkdirAll` (line 128) and `os.OpenFile` (line 144). Reachable via the public `RunUnzip`/`RunArchive` API wired to the user-facing unzip command. On Windows, `filepath.Join` also normalizes backslash separators, so `..\..\` traversal works. Absolute names like `/etc/passwd` are de-fanged by `Join` (treated as relative under `destDir`), but `..` traversal is fully effective. Trigger: extract an attacker-supplied `.zip` in CI/CD.
- **Recommended fix:** Add the same containment check as `archive-01` after computing `target` (`filepath.Clean` + `HasPrefix` against an absolute `destDir` + path separator), and reject absolute entry names. Do this before both the directory `MkdirAll` branch and the file-write branch.
- **Confidence:** high
- **Finding ID:** `archive-02`

#### [HIGH][security] Symlink extraction allows symlink-to-outside-dest plus follow-through write escape

- **File:** `internal/cli/archive/tar.go:205` (`extractTarArchive`)
- **CWE:** CWE-59
- **What/Why:** For `tar.TypeSymlink` the code creates `os.Symlink(header.Linkname, target)` without validating `header.Linkname`. An attacker can ship a symlink entry pointing at an absolute or `../`-relative location outside `destDir` (e.g. `Linkname="/etc"` or `"../../"`), then a later regular-file entry whose path traverses that symlink causes the `io.Copy`/`OpenFile` to write through the symlink to an arbitrary location (archive symlink-escape, CVE-2019-19921 class).
- **Evidence:**
  ```go
  case tar.TypeSymlink:
      if err := os.MkdirAll(filepath.Dir(target), 0755); err != nil {
          return err
      }
      if err := os.Symlink(header.Linkname, target); err != nil {
          return err
      }
  ```
- **Exploitability:** Realistic. Trigger: `omni tar -x -f attacker.tar(.gz) [-C dest]` on an attacker-crafted archive — a normal CI/dev action. Two attack variants work because there is zero containment validation: (1) Direct traversal via `header.Name`; (2) Symlink follow-through — entry A is a symlink (`Linkname="/tmp/evil"` or `"../../etc"`), then entry B is a regular file whose path goes through A, and `io.Copy` writes through the symlink (classic TOCTOU symlink escape). No special privileges needed. High rather than critical because it requires the user to choose to extract the malicious archive.
- **Recommended fix:** Validate that the resolved symlink destination stays within `destDir` before creating it; reject absolute `Linkname` outright. Additionally, before writing any regular file, verify no parent path component is an existing symlink (use a no-follow open such as `O_NOFOLLOW` on the final component, or `lstat` each parent), so later entries cannot be written through an attacker-planted symlink.
- **Confidence:** high
- **Finding ID:** `archive-03`

#### [HIGH][security] Hardlink extraction joins untrusted Linkname allowing hardlink to arbitrary file

- **File:** `internal/cli/archive/tar.go:213` (`extractTarArchive`)
- **CWE:** CWE-59
- **What/Why:** For `tar.TypeLink` the link target is computed as `filepath.Join(destDir, header.Linkname)` with no Clean/containment check, then `os.Link(linkTarget, target)` is called. `header.Linkname` is attacker-controlled and may contain `../` segments (e.g. `../../../../etc/shadow`), so the join escapes `destDir` and creates a hardlink to a sensitive file outside the extraction directory.
- **Evidence:**
  ```go
  case tar.TypeLink:
      ...
      linkTarget := filepath.Join(destDir, header.Linkname)
      if err := os.Link(linkTarget, target); err != nil {
          return err
      }
  ```
- **Exploitability:** Realistic. An attacker crafts a tar containing a `TypeLink` entry with `Linkname="../../../../etc/passwd"` plus an unvalidated `target` name. On `omni tar -x -f evil.tar`, `os.Link` creates a hardlink outside `destDir`. Combined with the unvalidated `target` (`archive-01`), the attacker controls where the link/file lands, enabling arbitrary out-of-tree link planting and TOCTOU/overwrite chains. A bare hardlink does not itself exfiltrate contents, which keeps this below the `TypeReg` write path in impact.
- **Recommended fix:** Apply containment checks to BOTH `target` and `linkTarget`: clean each, ensure each has the absolute `destDir`+separator prefix, and reject otherwise. Reject absolute `Linkname`. Only create the hardlink when both endpoints are provably inside `destDir`.
- **Confidence:** high
- **Finding ID:** `archive-04`

#### [HIGH][security] buf generate spawns external protoc process, violating the NO-EXEC invariant

- **File:** `internal/cli/buf/generate.go:116` (`runLocalPlugin`)
- **CWE:** CWE-78
- **What/Why:** `generate.go` imports `os/exec` and runs an external process: `exec.LookPath("protoc")` (line 91) and `exec.Command(protocPath, args...).Run()` (line 116). This is hand-written production code reachable from a live cobra command (`cmd/buf.go:206` → `buf.RunGenerate` → `runLocalPlugin` when a `buf.gen.yaml` plugin has a `local` entry). It violates omni's foundational NO-EXEC rule; the protoc arg list is assembled from `buf.gen.yaml` `local` plugin name/`opt` values, giving an argument-injection avenue into the spawned process.
- **Evidence:**
  ```go
  protocPath, err := exec.LookPath("protoc")
  ...
  cmd := exec.Command(protocPath, args...)
  cmd.Dir = dir
  cmd.Stdout = w
  cmd.Stderr = w
  return cmd.Run()
  ```
- **Exploitability:** Not realistically an attacker-triggered RCE. Args are passed via `exec.Command(protocPath, args...)` (not a shell), so there is no shell-metachar injection, and inputs come from a config file the operator authors. Trigger: any `omni buf generate` with a local plugin entry.

Either (a) drop local-plugin execution and emit the same "requires the full buf tool" notice used for remote plugins, or (b) implement code generation in pure Go against the already-built `FileDescriptorSet` (`build.go` already produces one via protocompile). If exec is truly unavoidable, gate it behind an explicit non-default build tag.
- **Confidence:** high
- **Finding ID:** `parsing-01`

#### [HIGH][security] internal/cli/exec spawns external processes via os/exec — NO-EXEC invariant violation

- **File:** `internal/cli/exec/exec.go:89` (`execute`)
- **CWE:** CWE-78
- **What/Why:** The exec package imports `os/exec` (aliased `osexec`) and runs arbitrary user-supplied commands via `osexec.Command(command, args...).Run()` with the child wired to `os.Stdin/Stdout/Stderr`. This directly violates omni's NO-EXEC principle. The command IS wired into the shipping CLI (`cmd/exec.go` imports `internal/cli/exec`), so it is live production behavior. The credential-detection pre-flight is a soft gate only: with `--force`, after the `[y/N]` prompt, or when no detector matches, the raw command executes with full inherited stdio.
- **Evidence:**
  ```go
  func execute(command string, args []string) error {
      cmd := osexec.Command(command, args...)
      cmd.Stdout = os.Stdout
      cmd.Stderr = os.Stderr
      cmd.Stdin = os.Stdin
      return cmd.Run()
  }
  ```
- **Exploitability:** No security exploit primitive. The command, args, and stdio are all operator-supplied; there is no injection point and no untrusted-data-to-exec flow, so CWE-78 does not apply in the classic sense. The realistic trigger for the underlying defect is simply that the command exists and runs in the default binary, contradicting the NO-EXEC design principle. It is a robustness/invariant defect.
- **Recommended fix:** Remove the `os/exec` dependency to restore NO-EXEC. Either delete the exec command entirely or repurpose it as a pure-Go, no-spawn credential pre-flight checker that only PRINTS what would be needed. If a launcher is genuinely required, gate it behind a separate, clearly-named build tag excluded from the default binary and document the exception. Do not silently keep `os/exec` in the default build.
- **Confidence:** high
- **Finding ID:** `process-01`

#### [HIGH][supply-chain] forloop spawns sh -c / cmd /C with interpolated user input (no-exec violation + command injection)

- **File:** `internal/cli/forloop/forloop.go:222` (`executeCommand`)
- **CWE:** CWE-78
- **What/Why:** The forloop command (`omni for range/each/lines/split/glob`) builds a command string by interpolating loop variables (range index, list items, file lines, glob matches) into a user-supplied template via `replaceVariable`, then executes it through the system shell: `exec.Command("sh", "-c", command)` on Unix and `exec.Command("cmd", "/C", command)` on Windows. This violates NO-EXEC and, because loop values flow unescaped into the shell string, hostile filenames or stdin lines (e.g. a line containing `; rm -rf ~`) yield arbitrary command execution.
- **Evidence:**
  ```go
  func executeCommand(w io.Writer, command string) error {
      var cmd *exec.Cmd
      if isWindows() {
          cmd = exec.Command("cmd", "/C", command)
      } else {
          cmd = exec.Command("sh", "-c", command)
      }
  ```
- **Exploitability:** Realistic and low-effort. NO-EXEC breach triggers on any non-dry-run invocation. Injection: `omni for lines log.txt -- echo $line` where a line contains `; rm -rf ~`, or `omni for glob "*" -- echo $file` over a directory containing a file named `$(rm -rf ~)`, interpolates unescaped into `sh -c` and executes. `lines`/`glob` pull attacker-controllable data. The binary also fails on any host lacking `sh`/`cmd.exe`.
- **Recommended fix:** Remove `os/exec` from forloop. Either (a) restrict `for` to driving omni's own internal Command registry by tokenizing the template and dispatching in-process, or (b) drop the execute path and keep only `--dry-run`/template-expansion output. If external execution is ever required it must be an opt-in, argv-array (never shell-string) call gated behind an explicit flag.
- **Confidence:** high
- **Finding ID:** `supply-01`

#### [HIGH][supply-chain] task ShellCommandRunner/HybridCommandRunner execute via system shell (no-exec violation + injection)

- **File:** `internal/cli/task/cmdrunner.go:125` (`ShellCommandRunner.Run`)
- **CWE:** CWE-78
- **What/Why:** `ShellCommandRunner.Run` joins args into a single string and runs it through `exec.CommandContext(ctx, "cmd", "/C", cmdStr)` / `("sh", "-c", cmdStr)`. `HybridCommandRunner` routes any command Cobra reports as "unknown command" to this shell runner as a fallback. This spawns external processes (NO-EXEC violation, host shell becomes an undeclared runtime dependency) and, because args are concatenated into one shell string, is injectable. It also selects the shell via `runtime.GOOS == "windows"` rather than a build tag.
- **Evidence:**
  ```go
  cmdStr := strings.Join(args, " ")
  var cmd *exec.Cmd
  if runtime.GOOS == "windows" {
      cmd = exec.CommandContext(ctx, "cmd", "/C", cmdStr)
  } else {
      cmd = exec.CommandContext(ctx, "sh", "-c", cmdStr)
  }
  ```
- **Exploitability:** Reachable in production. With `omni task --allow-external`, the factory returns `NewHybridCommandRunner`; the executor calls `cmdRunner.Run` for every task command, and on an "unknown command" error falls back to `ShellCommandRunner.Run`. Trigger: any Taskfile.yml command whose first word is not a known omni subcommand, with variable/env interpolation as the injection sink. An untrusted-repo Taskfile makes it a genuine RCE vector. The runners have 0.0% test coverage, confirming an unexercised escape hatch.
- **Recommended fix:** Eliminate the shell fallback. The task runner already has `CobraCommandRunner` for in-process dispatch; make "unknown command" a hard error (`cmderr.ErrUnsupported`) instead of falling back to `sh`/`cmd`, and delete `ShellCommandRunner`/`HybridCommandRunner`.
- **Confidence:** high
- **Finding ID:** `supply-02`

#### [MEDIUM][robustness] Index-out-of-range panic in ls -a when directory Lstat fails but parent Lstat succeeds

- **File:** `internal/cli/ls/ls.go:169` (`listPath`)
- **CWE:** CWE-129
- **What/Why:** In the `-a` branch, the `.` entry is only prepended when `os.Lstat(path)` succeeds. If that Lstat fails (TOCTOU removal after ReadDir) AND the directory has zero visible entries, `entries` is empty. The subsequent `os.Lstat(parentPath)` can still succeed, and line 169 then evaluates `entries[0]` / `entries[1:]` on an empty slice, causing an index-out-of-range panic that crashes the process instead of returning a classified error.
- **Evidence:**
  ```go
  if dotInfo, err := os.Lstat(path); err == nil { dot := ...; entries = append([]Entry{dot}, entries...) }
  parentPath := filepath.Dir(path)
  if parentInfo, err := os.Lstat(parentPath); err == nil {
      dotdot := fileInfoToEntry(parentPath, parentInfo)
      dotdot.Name = ".."
      entries = append([]Entry{entries[0], dotdot}, entries[1:]...)  // entries[0] panics if entries is empty
  }
  ```
- **Exploitability:** TOCTOU / transient-filesystem edge: the path is removed after `ReadDir` succeeds but before `Lstat`, leaving an empty `entries` while parent `Lstat` still succeeds. Crashes `omni ls -a` rather than returning a classified error.
- **Recommended fix:** Guard the `..` prepend on `len(entries) > 0`, or build the `.`/`..` entries into a separate prefix slice and prepend once (`entries = append(prefix, entries...)`). Do not index `entries[0]` without a length check.
- **Confidence:** high
- **Finding ID:** `fileops-01`

#### [MEDIUM][robustness] dd block-size / count parsing has no overflow or non-positive validation, enabling make([]byte) panic or OOM

- **File:** `internal/cli/dd/dd.go:153` (`RunDd` / `ParseDdSize`)
- **CWE:** CWE-190
- **What/Why:** `ParseDdSize` computes `num * multiplier` with no overflow check, so e.g. `bs=20000000000000T` wraps int64 to a negative/absurd value. `RunDd` never validates that `InputBS`/`OutputBS` are strictly positive before `make([]byte, opts.InputBS)` (line 153) and `make([]byte, 0, opts.OutputBS)` (line 154). A negative size triggers a `makeslice: len out of range` panic; a huge positive size causes an unbounded allocation / OOM. The reslice loop `outBuf = outBuf[opts.OutputBS:]` (line 191) also misbehaves if `OutputBS <= 0`.
- **Evidence:**
  ```go
  buf := make([]byte, opts.InputBS)   // panics if InputBS <= 0; OOM if huge
  ...
  return num * multiplier, nil        // ParseDdSize: no overflow check on num*multiplier
  ```
- **Exploitability:** Triggerable purely from CLI operands. `omni dd ibs=20000000000000T` wraps the product to a negative `InputBS`, reaching `make([]byte, opts.InputBS)` and panicking with no `recover()` in the call chain (process crash). Negative `OutputBS` makes the loop guard always true and `outBuf[opts.OutputBS:]` panics. Even without overflow, `bs=5000000000` attempts a ~5GB allocation (OOM). Realistic trigger is a malformed operand, so impact is a crash/OOM of the invoking process.
- **Recommended fix:** After defaulting, validate `opts.InputBS > 0 && opts.OutputBS > 0` and return `cmderr.ErrInvalidInput` otherwise. In `ParseDdSize`, detect multiplication overflow (`num != 0 && result/num != multiplier`) and reject negative/zero results. Optionally cap block size to a sane maximum.
- **Confidence:** high
- **Finding ID:** `fileops-02`

#### [MEDIUM][security] Unbounded io.ReadAll of remote HTTP bodies enables memory-exhaustion / decompression-bomb DoS

- **File:** `pkg/video/nethttp/client.go:158` (`Client.GetString` / `GetJSON` / `PostJSON`)
- **CWE:** CWE-409
- **What/Why:** `GetString` (158), `GetJSON` (179) and `PostJSON` (205) call `io.ReadAll(resp.Body)` on attacker-controlled responses (YouTube player JS, InnerTube JSON, arbitrary webpages, M3U8 manifests, AES key bodies) with no `io.LimitReader` or max-size guard. The default `http.Transport` leaves automatic gzip decompression enabled, so a malicious server can serve a small gzip body that net/http transparently inflates into gigabytes, exhausting memory.
- **Evidence:**

  ```go
  body, err := io.ReadAll(resp.Body)  // GetString, no size cap; same in GetJSON/PostJSON
  ```
- **Exploitability:** Realistic but self-inflicted: triggered when the user passes a malicious/compromised http(s) URL to the video downloader (the generic extractor accepts ANY http/https URL), or when a legitimate host is MITM'd. `NewClient`'s transport never sets `DisableCompression`, so a small compressed payload can expand to gigabytes and crash the process. The same unbounded `io.ReadAll` exists in `hls.go:179` (segments) and `hls.go:197` (AES key bodies). The 30s client `Timeout` bounds slow-loris hangs but not a fast, large/decompressed body. Client-side DoS against the operator.
- **Recommended fix:** Wrap `resp.Body` in `io.LimitReader` with a sane ceiling (e.g. 64–128 MB for HTML/JSON/manifests) before `io.ReadAll`, and treat hitting the limit as an error. Cap player JS/JSON tightly. Consider disabling transparent decompression and bounding decompressed size explicitly.
- **Confidence:** high
- **Finding ID:** `netvideo-01`

#### [MEDIUM][security] goja runs untrusted YouTube player JS with no interrupt, timeout, or stack/step limit (DoS)

- **File:** `pkg/video/jsinterp/jsinterp.go:24` (`Interpreter.Execute` / `CallFunction` / `ExtractFunction`)
- **CWE:** CWE-400
- **What/Why:** `jsinterp.New()` creates a bare `goja.Runtime` and `Execute`/`CallFunction`/`ExtractFunction` call `vm.RunString` / `fn(...)` directly on JavaScript scraped at runtime from YouTube's remote player JS. No `vm.Interrupt` watchdog, no context deadline, and no `SetMaxCallStackSize`. A malicious or compromised player script can run an infinite loop and hang the omni process indefinitely, since goja executes synchronously and cannot be cancelled without a pre-armed Interrupt.
- **Evidence:**
  ```go
  vm := goja.New() ... v, err := i.vm.RunString(code)  // no Interrupt/deadline/stack limit
  ```
- **Exploitability:** Triggered only on the opt-in YouTube download path when formats need signature/nsig decryption. - **Recommended fix:** Quote SQL identifiers correctly: reject names containing a NUL byte, then wrap in double quotes and double any embedded double-quote. Centralize in a `quoteIdent(name string) string` helper and use it everywhere an identifier is interpolated (lines 165, 265, 655, 691). Do not use `%q` for SQL identifiers.
- **Confidence:** high
- **Finding ID:** `db-01`

#### [LOW][robustness] Unbounded allocation from attacker-controlled page size in malicious bbolt file

- **File:** `internal/cli/bbolt/bbolt.go:646` (`RunPageDump`)
- **CWE:** CWE-789
- **What/Why:** `RunPageDump` reads the page size from bytes 12..15 of an untrusted `.db` file as a little-endian uint32 and then allocates `make([]byte, pageSize)` directly. A crafted bbolt file can set these bytes to a value near 2^31-1, forcing an immediate ~2 GB heap allocation (memory-exhaustion DoS) on a single `omni bbolt page-dump file 0` invocation. The only guard is `pageSize == 0` → 4096; there is no upper bound.
- **Evidence:**
  ```go
  pageBuf := make([]byte, pageSize)

  n, err := f.Read(pageBuf)
  ```
- **Exploitability:** Realistically triggerable: an attacker who can get a victim to run `omni bbolt page <crafted.db> 0` on a file whose bytes 12-15 encode a large value (e.g. `0xFFFFFFFF`) forces an immediate ~4 GB heap allocation. No privilege or network needed — just file delivery, which matches omni's untrusted-file-processing scope. Effect is a single-process memory spike that may OOM-panic that one run; it does not persist or corrupt state. Cannot be made negative (64-bit int), so no out-of-bounds path.
- **Recommended fix:** After parsing `pageSize`, validate it against the bbolt-legal range (a power of two between 512 and 65536) and clamp it to the actual file size before allocating: `if pageSize < 512 || pageSize > 65536 { return cmderr.Wrap(cmderr.ErrInvalidInput, "bbolt: invalid page size") }`. Reject oversized pageID offsets likewise.
- **Confidence:** high
- **Finding ID:** `db-02`

#### [LOW][security] gopsagent exposes profiling, GC-tuning and shutdown opcodes unauthenticated when AuthKey is unset

- **File:** `pkg/gopsagent/handlers.go:31` (`Agent.handle`)
- **CWE:** CWE-306
- **What/Why:** `handle()` runs the HMAC challenge only when `len(a.opts.AuthKey) > 0`. With the default `Options{}` (AuthKey nil) the agent accepts every opcode with no authentication: `OpShutdown` (when `AllowShutdown` is set), `OpSetGCPercent` (attacker-chosen GC value), `OpCPUProfile`/`OpTrace`/`OpHeapProfile` (pull profiling data, tie up global profilers), and `OpStack` (dumps full goroutine stacks, potentially leaking sensitive args/paths). The README example starts the agent with `gopsagent.New(gopsagent.Options{})` — the insecure-by-default configuration.
- **Evidence:**
  ```go
  func (a *Agent) handle(conn net.Conn) {
      defer func() { _ = conn.Close() }()
      if len(a.opts.AuthKey) > 0 {
          if !runAuthChallenge(conn, a.opts.AuthKey) {
              return
          }
      }
      op := make([]byte, 1)
      if _, err := io.ReadFull(conn, op); err != nil {
          return
      }
      switch op[0] {
      ...
      case OpSetGCPercent:
          var pct int
          _, _ = fmt.Fscanf(conn, "%d", &pct)
          debug.SetGCPercent(pct)
      ...
      case OpShutdown:
          if a.opts.AllowShutdown {
              go func() { _ = a.Close() }()
          }
      }
  ```
- **Exploitability:** Requires the embedding program to have called `Listen()` (opt-in; only call sites are tests) and a malicious local process able to discover the random loopback port. The pid file is mode 0600/dir 0700 so a different-UID co-tenant cannot read it and must port-scan 127.0.0.1. Once connected, with default `Options{}` an attacker gets info disclosure (`OpStack`) and a GC-tuning DoS (`OpSetGCPercent`). Process termination is NOT reachable unless `AllowShutdown` was explicitly enabled. Not remotely exploitable by default (loopback bind).
- **Recommended fix:** Treat state-changing opcodes (`OpShutdown`, `OpSetGCPercent`, `OpCPUProfile`, `OpTrace`, `OpHeapProfile`, `OpStack`) as privileged: require `AuthKey` to be set before dispatch. Prefer making `AuthKey` mandatory (return an error from `Listen` when empty unless an explicit `AllowInsecure` flag is set), mirroring the existing `AllowNonLoopback` opt-in pattern, so the default is closed.
- **Confidence:** high
- **Finding ID:** `process-03`

#### [LOW][robustness] free: unsigned integer underflow when computing used memory/swap

- **File:** `internal/cli/free/free.go:70` (`RunFree`)
- **CWE:** CWE-191
- **What/Why:** `memUsed` and `swapUsed` are computed by subtracting uint64 values without guarding against underflow. If the source data is inconsistent — e.g. a `/proc/meminfo` read that captured MemFree/Buffers/Cached but not MemTotal (partial read leaves MemTotal == 0) — the subtraction wraps around to an enormous bogus value (~1.8e19) that is then divided and printed, producing nonsensical output instead of a clear error.
- **Evidence:**
  ```go
  memUsed := info.MemTotal - info.MemFree - info.Buffers - info.Cached
  swapUsed := info.SwapTotal - info.SwapFree
  ```
- **Exploitability:** No realistic attacker trigger; data sources are kernel-synthesized, not user-controlled. The only realistic (non-attacker) trigger is an unchecked `scanner.Err()`/partial read of `/proc/meminfo` on Linux leaving MemTotal==0, producing a wrapped ~1.8e19 value in printed output. Very low likelihood, cosmetic impact, no crash/data loss.
- **Recommended fix:** Clamp each subtraction with a saturating helper, e.g. `func sub(a, b uint64) uint64 { if b > a { return 0 }; return a - b }` and chain it, or validate that `MemTotal >= MemFree+Buffers+Cached` before computing and otherwise report `cmderr.ErrIO`. Apply the same guard to `swapUsed`.
- **Confidence:** medium
- **Finding ID:** `process-05`

#### [LOW][supply-chain] internal/cli/exec spawns arbitrary external commands (no-exec violation)

- **File:** `internal/cli/exec/exec.go:88` (`execute`)
- **CWE:** CWE-749
- **What/Why:** The exec package runs an arbitrary command/args supplied by the user via `osexec.Command(command, args...)` wired to `os.Stdin/Stdout/Stderr`. The package's own doc comment acknowledges this is 'a pre-existing violation of the project "No exec" design principle.' Shipping this in the omni binary contradicts the foundational invariant and makes whatever the user names a transitive, undeclared dependency.
- **Evidence:**
  ```go
  // NOTE: This package intentionally spawns an external process via os/exec.
  // This is a pre-existing violation of the project "No exec" design principle.
  ...
  func execute(command string, args []string) error {
      cmd := osexec.Command(command, args...)
  ```
- **Exploitability:** No attacker-controlled trigger. `omni exec` takes the command and args directly from argv; spawning that process is the documented purpose, not an injection. There is no path where untrusted file/archive/network input flows into `osexec.Command`. The CWE-749 / supply-chain framing of injection does not fit. The only genuine concern is architectural: it violates omni's foundational NO-EXEC invariant and ships in the default binary with no build-tag gate (`cmd/exec.go` registers `execCmd` unconditionally).
- **Recommended fix:** Either remove the `omni exec` command from the shipped binary, or move it behind an explicit, separately-documented build tag (e.g. `//go:build omni_unsafe_exec`) so default builds cannot spawn processes. Track and execute the remediation referenced in deferred-items.md rather than leaving it shipping by default.
- **Confidence:** high
- **Finding ID:** `supply-03`

## Resolution status

This section records the disposition of the no-exec / supply-chain findings after the no-exec boundary decision was documented. See `docs/architecture/patterns.md` § "No-exec invariant: scope & sanctioned exceptions" for the governing rule.

### ACCEPTED — sanctioned wrapper, documented

These findings flag `os/exec` usage in commands whose **purpose** is to orchestrate an external tool. They fall under the sanctioned-exception set in the no-exec invariant subsection and are accepted as-is (kept argv-only, injection-safe). They are NOT NO-EXEC violations once scoped to "utility reimplementations":

- **`parsing-01`** — `buf generate` local-plugin path drives `protoc` / local codegen plugins. ACCEPTED (sanctioned wrapper, documented).
- **`process-01`** — `internal/cli/exec` launches an operator-supplied command; the launcher is the feature. ACCEPTED (sanctioned wrapper, documented).
- **`supply-01`** — `forloop` runs a per-iteration command template. ACCEPTED (sanctioned wrapper) **and INJECTION-HARDENED**: the auto-generated loop value is no longer string-interpolated into the shell command. It is passed via the child process environment and expanded by the shell as an inert single token — POSIX `sh` uses native `$var` expansion of the env value; Windows `cmd` is invoked with `/V:ON` and delayed-expansion `!var!` so the value expands *after* tokenization and metacharacters (`&`,`|`,`>`,`;`,`$()`) cannot inject commands. Pinned by `TestForLoop_InjectionSafe` (POSIX + Windows vectors).
- **`supply-02`** — `task` command runners. ACCEPTED (sanctioned wrapper) **and INJECTION-HARDENED**: the single-string shell-command feature is preserved, but multi-arg invocations no longer concatenate into a re-parseable shell string. POSIX uses `sh -c 'exec "$0" "$@"' prog args…` (positional params); Windows passes non-leading args via the environment referenced with delayed-expansion `!OMNI_ARGn!` under `cmd /V:ON /C`. Pinned by `TestTaskRunner_InjectionSafe` (incl. the space-free Windows vectors that exposed the original gap).
- **`supply-03`** — `internal/cli/exec` build-tag/consolidation item. ACCEPTED (sanctioned wrapper, documented).
- **`supply-04`** — `terraform` (`omni tf`) wraps the `terraform` binary. ACCEPTED (sanctioned wrapper, documented) — external prerequisite.
- **`supply-05`** — git hacks / gh hacks drive the `git` and `gh` binaries. ACCEPTED (sanctioned wrapper, documented) — argv-only, IDs parsed as ints.
- **`supply-06`** — `repo` (remote clone) drives `git` / `gh`. ACCEPTED (sanctioned wrapper, documented) — argv invocation only.

### DEFERRED

- **`crypto-02`** — Raising `DefaultIter` toward 600000 cannot be done safely yet: the PBKDF2 iteration count is NOT stored in the ciphertext envelope, so changing the default would make existing on-disk blobs (encrypted with the old count) undecryptable. DEFERRED — requires a versioned ciphertext envelope that persists the iteration count before the default can be raised. Track as a deprecation task (introduce versioned envelope → dual-read old/new → raise default → cleanup after deprecation window). The companion floor-validation (`crypto-01`) WAS applied.
- **`cloud-01` (darwin half)** — `machineid_darwin` still parses the macOS hardware UUID (`IOPlatformUUID`) from `ioreg`. macOS exposes this value only via IOKit (cgo) or the `ioreg` CLI; there is no pure-Go/no-cgo API that returns the **same** identifier, and the machine ID feeds the master-key PBKDF2 in `master.go`, so changing its value would make every existing machine-bound `master.key` undecryptable. DEFERRED — `ioreg` is retained as the single **documented** machine-identity exec exception (treated like a sanctioned platform-API wrapper). Removing it requires either a pure-Go-reachable `IOPlatformUUID` source or a `master.key` re-encryption migration on identifier change. The Windows half (`uname_windows`) was reimplemented — see below.

### PURE-GO-REIMPLEMENTED

This platform helper was *not* a sanctioned wrapper (it reimplements a value-derivation that has a pure-Go equivalent), so it was reimplemented to remove the subprocess while staying build-tagged:

- **`supply-07` (windows half)** — `uname_windows` no longer spawns `cmd /c ver`. PURE-GO-REIMPLEMENTED via `golang.org/x/sys/windows` `RtlGetVersion`, preserving the `//go:build windows` tag and the existing output format/signature.
