### cmdinjection/task-shellrunner-windows-leading-element

- **Title:** Windows command injection via RAW leading argv element in task ShellCommandRunner multi-arg branch
- **Location:** `internal/cli/task/cmdrunner.go:181`  ·  **CWE:** CWE-78: OS Command Injection  ·  **Verdict:** 3/3 lenses, reproduced=True  ·  **Fix invariant-safe:** True
- **What:** The supply-02 fix neutralizes cmd.exe metacharacters in NON-leading argv elements by binding each to an OMNI_ARGn env var and referencing it via delayed expansion (`!OMNI_ARGn!`). However the LEADING element is written verbatim into the cmd.exe command string: `b.WriteString(args[0])` (line 181), producing `cmd /V:ON /C <args[0]> !OMNI_ARG0!...`. cmd.exe parses `& | < > ^` in args[0] BEFORE delayed expansion, so a leading element containing those metacharacters is reparsed as command syntax. The code comment on lines 172-173 explicitly (and wrongly) claims the leading element is safe ('kept as a literal token so it still resolves as a command'); a literal token like `x&calc` is NOT inert. The POSIX branch is unaffected because it uses `sh -c 'exec "$0" "$@"' prog args...`, where the leading element is bound to `$0` and not reparsed (verified). Reachability: executor.go does `cmdStr := resolver.Expand(cmd.Cmd)` (interpolating untrusted Taskfile vars/env into the command string) then `parseCommand(cmdStr)` splits on whitespace; if an interpolated value lands at the START of the command with no preceding space and the first token is not an omni builtin, with `--allow-external` the HybridCommandRunner routes the 'unknown command' to ShellCommandRunner where it becomes args[0]. The runners have 0% test coverage.
- **Exploit:** Untrusted Taskfile.yml run with `omni task --allow-external build`:
```yaml
version: '3'
vars:
  PKG: "x&calc"
tasks:
  build: { cmds: ["{{.PKG}} install"] }
```
Expand -> `x&calc install`; parseCommand -> ["x&calc","install"]; firstWord `x&calc` is not an omni builtin -> Hybrid -> ShellCommandRunner multi-arg branch -> `cmd /V:ON /C x&calc !OMNI_ARG0!`. cmd.exe reparses the `&` and launches calc. Empirically proven: cmdline `[x&echo.>MARKER !OMNI_ARG0!]` created the marker file and a `[echoFIRST&echo SECOND_CMD_RAN ...]` payload executed the chained `echo SECOND_CMD_RAN`. A space-free `&`-chained payload in the leading token achieves arbitrary command execution; even on a trusted Taskfile, an env/var value flowing into the first token (e.g. a tool name resolved from `$TOOL`) is an injection sink.
- **Fix:** Treat the leading element identically to the others on Windows: never concatenate any argv element raw into the cmd.exe string. Bind args[0] to an env var too (e.g. OMNI_ARG_PROG) and reference it via delayed expansion: build `cmd /V:ON /C !OMNI_ARG_PROG! !OMNI_ARG0! !OMNI_ARG1!...`. This keeps the whole pipeline pure-Go/no-CGO and cross-platform (POSIX branch unchanged, already safe). Note the existing documented limitation: delayed expansion mangles a literal `!` in any value, including the program name — acceptable per the project's stated tradeoff. Add a Windows-only table-driven test asserting that a leading token containing `&|<>^` is NOT reparsed (RED->GREEN against the PoC above).

### verifysoundness/scan-failon-unknown-severity-bypass

- **Title:** --fail-on CI gate silently ignores findings whose severity is "unknown" (no CVSS data)
- **Location:** `pkg/scan/scan.go:80`  ·  **CWE:** CWE-755 Improper Handling of Exceptional Conditions  ·  **Verdict:** 3/3 lenses, reproduced=True  ·  **Fix invariant-safe:** True
- **What:** The --fail-on gate in scanComponents only enters its loop when opts.FailOn > SeverityUnknown, and then trips only when ParseSeverity(f.Severity) >= opts.FailOn. A finding whose OSV record carries no parseable CVSS_V3/CVSS_V4 score gets severityLabel()==SeverityUnknown -> f.Severity=="unknown". ParseSeverity("unknown") returns (SeverityUnknown=0, false), and 0 is never >= any real threshold (low..critical are 2..5). So a genuine, matched vulnerability with no CVSS vector ALWAYS passes the gate. This is not theoretical: a large fraction of Go advisories (GO-YYYY-NNNN) ship without a CVSS vector, and OSV severity[] is frequently absent or v4-only-without-numeric (severity.go only computes scores for CVSS_V3, so a v4-only record with a non-numeric vector also lands as unknown). Empirically reproduced against the real pkg/scan: an entry with no severity[] matched a component, produced findings=1, yet scanComponents(...,FailOn:SeverityLow) returned gate-err=nil. The whole point of --fail-on in CI is to fail the build on vulnerable deps; a real vuln slips through with exit 0.
- **Exploit:** A CI pipeline runs `omni scan sbom.json --db osv-db.zip --db-key k.pub --fail-on critical` (or any threshold). A dependency in the SBOM is hit by a signed-DB OSV entry that has no CVSS_V3/V4 numeric score (very common for GO-prefixed advisories). The finding is reported in the table but the command exits 0, so the merge/release proceeds with a known-vulnerable, unfixed dependency. No attacker is required, only an ordinary uncscored advisory; an attacker who knows their vuln has no CVSS vector can rely on the gate staying green.
- **Fix:** Decouple the gate decision from the rendered string. Carry the Severity enum on Finding (e.g. add a non-serialized sevRank Severity field set at match time from severityLabel) and gate on that enum, not on re-parsing f.Severity. Then decide policy for unknown explicitly: the safe default for a CI gate is fail-closed -- treat SeverityUnknown as tripping any active --fail-on (or add an explicit --fail-on-unknown=false opt-out). At minimum, when opts.FailOn is set and a finding is SeverityUnknown, trip the gate (or surface a distinct non-zero exit) rather than silently passing. Pure-Go, no new deps, cross-platform.

### cmdinjection/repo-git-clone-argument-injection

- **Title:** Argument injection into git/gh clone: untrusted target starting with '-' reaches git as a flag (no -- option terminator)
- **Location:** `internal/cli/repo/remote.go:48`  ·  **CWE:** CWE-88: Argument Injection  ·  **Verdict:** 3/3 lenses, reproduced=True  ·  **Fix invariant-safe:** True
- **What:** cloneToTemp builds `git clone --depth=1 <cloneURL> <tmpDir>` (line 48) and `gh repo clone <cloneURL> <tmpDir> -- --depth=1` (line 25) where cloneURL = normalizeRemote(target) and target is the `omni repo analyze <target>` operand. normalizeRemote does not reject or neutralize a leading '-', and neither clone invocation places cloneURL after a `--` end-of-options marker (the gh `--` on line 25 is positioned AFTER the positionals, so it guards nothing on the URL). isRemote() can classify a target containing a '/' as remote (owner/repo shorthand), and a value like `-x/y` passes `!strings.HasPrefix(target,"/")` and the SplitN==2 check, so it routes to cloneToTemp. A cloneURL beginning with '-' is then interpreted by git/gh as an option rather than a URL. git's `ext::`/`fd::` transport RCE is de-fanged here because non-http/git@ targets get `https://github.com/` prepended, but option-injection flags (e.g. `--upload-pack=`, `-c protocol.ext.allow=...`, `--config`) remain a real surface, and clone of `-`-prefixed names mis-parses. This is argv-array exec (no shell), so it is argument injection rather than classic OS command injection; impact is bounded by what git/gh flags can do at clone time.
- **Exploit:** `omni repo analyze -- '--upload-pack=touch /tmp/x/y'`-style target where the value passes isRemote() as `owner/repo` shorthand (contains one '/', no leading '/' or '.'), e.g. target `--config=core.fsmonitor=.../y`. normalizeRemote returns it unchanged (no http/git@ prefix => only TrimPrefix github.com/ and TrimSuffix .git apply), then for the git fallback it gets `https://github.com/` prepended ONLY if it does not start with https/git@ -> actually it IS prepended, de-fanging many cases; the residual exploitable case is the gh path (line 25) which passes cloneURL unprepended and unguarded, so a `-`-leading owner/repo reaches `gh repo clone -<flag> tmpDir`. An attacker who controls the analyze target (e.g. a CI job templating a repo name from PR metadata) injects a gh/git flag.
- **Fix:** Reject targets whose normalized cloneURL begins with '-' (return cmderr.ErrInvalidInput), and insert a `--` option terminator immediately before the URL in BOTH invocations: `git clone --depth=1 -- <cloneURL> <tmpDir>` and `gh repo clone -- <cloneURL> <tmpDir>` (gh) so the URL can never be parsed as a flag. Also validate the URL shape (host allowlist / scheme check) before clone. Pure-Go, cross-platform, no new deps; this is part of the already-sanctioned exec wrapper so no invariant change.

### resourcedos/zip-decompression-bomb-uncapped

- **Title:** Zip extraction has no decompression-bomb cap (archive-05 fix applied to tar only, not zip)
- **Location:** `internal/cli/archive/zip.go:175`  ·  **CWE:** CWE-409: Improper Handling of Highly Compressed Data (Data Amplification)  ·  **Verdict:** 3/3 lenses, reproduced=True  ·  **Fix invariant-safe:** True
- **What:** The archive-05 hardening fix added a per-entry io.LimitReader plus a cumulative `maxExtractTotalBytes` (10 GiB) counter to the tar regular-file extraction path (internal/cli/archive/tar.go:235-246), but the zip extraction path was NOT updated. `extractZipArchive` still does a bare `io.Copy(outFile, rc)` at zip.go:175 with no io.LimitReader, no per-entry cap, and no cumulative byte counter. `f.UncompressedSize64` is attacker-declared and is never used to bound the write. A small malicious .zip whose entries inflate to terabytes (a classic zip bomb, e.g. a deeply-nested or highly-redundant deflate stream) fills the CI runner's disk. The HARDENING.md recommended fix for archive-05 explicitly said 'Apply the same to the zip branch'; that was not done.
- **Exploit:** Attacker ships a crafted small .zip (e.g. a 42 KB nested zip bomb expanding to many GB, or a single deflate entry of ~10 MB that inflates to >100 GB). Victim runs `omni unzip evil.zip` (or `omni tar -x` on a .zip via RunArchive) in CI/CD over an untrusted artifact; extraction writes the inflated stream to disk with no cap, exhausting disk space and DoS-ing the runner.
- **Fix:** Mirror the tar.go fix in extractZipArchive: thread a `var totalWritten int64` through the loop and replace `io.Copy(outFile, rc)` with `n, err := io.Copy(outFile, io.LimitReader(rc, (maxExtractTotalBytes - totalWritten)+1))`, then `totalWritten += n; if totalWritten > maxExtractTotalBytes { return cmderr.Wrap(cmderr.ErrInvalidInput, "archive: extraction exceeds maximum allowed size") }`. Reuse the existing `maxExtractTotalBytes` const. Pure-stdlib (archive/zip, io), cross-platform, no new deps.

### resourcedos/hls-segment-unbounded-readall

- **Title:** HLS segment download uses unbounded io.ReadAll, bypassing the netvideo-01 readAllLimited cap
- **Location:** `pkg/video/downloader/hls.go:192`  ·  **CWE:** CWE-770: Allocation of Resources Without Limits or Throttling  ·  **Verdict:** 3/3 lenses, reproduced=True  ·  **Fix invariant-safe:** True
- **What:** The netvideo-01 fix added `readAllLimited` (128 MiB cap) and wired it into Client.GetString/GetJSON/PostJSON in pkg/video/nethttp/client.go. But `HLSDownloader.downloadSegment` does NOT go through those wrappers: it builds its own request, calls `opts.Client.Do(req)` directly, then `data, err := io.ReadAll(resp.Body)` at hls.go:192 with no io.LimitReader and no size cap. The segment URL (`segURL`) is derived from the untrusted M3U8 manifest. The 30s client Timeout bounds slow responses but not a fast, large body. Each segment is fully buffered in memory before being written, so a single oversized segment body is the unbounded allocation unit. (The SSRF guard validateFetchURL is applied, but it does not bound body size.)
- **Exploit:** Victim runs `omni video <hostile-or-MITM'd-HLS-URL>`. The attacker controls the manifest, lists a segment whose server streams a multi-gigabyte (or transparently gzip-inflated) body; `downloadSegment` calls `io.ReadAll(resp.Body)` and buffers the whole thing in RAM, exhausting memory and crashing the process.
- **Fix:** Route the segment read through the same cap: either call a new exported `Client.GetBytesLimited`/reuse `readAllLimited`, or inline `io.ReadAll(io.LimitReader(resp.Body, maxResponseBytes+1))` with the overflow check used in nethttp.readAllLimited (return an error, not a truncation). Segments legitimately can exceed 128 MiB for long HLS chunks, so pick a per-segment ceiling (e.g. 256-512 MiB) or stream-decrypt in fixed-size blocks. Pure-stdlib (io), cross-platform.

### verifysoundness/scan-exact-version-v-prefix-mismatch

- **Title:** Exact version-membership match uses raw string equality, so a v-prefix asymmetry between OSV versions[] and SBOM purl version silently misses vulnerable components
- **Location:** `pkg/scan/match.go:72`  ·  **CWE:** CWE-697 Incorrect Comparison  ·  **Verdict:** 2/3 lenses, reproduced=True  ·  **Fix invariant-safe:** True
- **What:** affectedHit's exact-membership shortcut compares `v == version` with plain string equality and NO normalization, while the SEMVER interval path normalizes both sides via sv() (prepend "v"). purl.Parse preserves the version verbatim, so an SBOM purl `pkg:golang/m@v1.2.3` yields version "v1.2.3" and `@1.2.3` yields "1.2.3". OSV versions[] entries for Go are stored in whichever convention the producer used. When the prefix presence differs, the exact match fails. Confirmed empirically: OSV versions=["1.2.3"] vs SBOM version "v1.2.3" -> hit=false (miss); vs "1.2.3" -> hit=true. The code's own comment states ECOSYSTEM and GIT range types are "handled by exact versions only" -- meaning EVERY advisory whose ranges are ECOSYSTEM/GIT (not SEMVER) depends entirely on this un-normalized exact path, so a v-prefix mismatch makes the entire advisory unmatchable for that component. Result: a real, signed-DB vulnerability is reported as not-present (fail-open under-report), and combined with the --fail-on gate the CI build stays green.
- **Exploit:** An SBOM produced by a tool that emits v-prefixed Go purl versions (e.g. `pkg:golang/github.com/x/y@v1.2.3`) is scanned against a signed OSV DB whose advisory for that module lists the affected version in versions[] as "1.2.3" (no v) under an ECOSYSTEM/GIT range. matchEntry returns ok=false, the vulnerable component is omitted from findings entirely, and `omni scan --fail-on high` exits 0. The reverse (OSV "v1.2.3" vs SBOM "1.2.3") misses too.
- **Fix:** Normalize both operands in the exact-membership loop the same way the interval path does: compare sv(v) == sv(version) (and also try the raw == for non-semver ECOSYSTEM tags as a fallback). Equivalently, canonicalize component versions and OSV versions[] to a single convention (strip-or-add the leading v) at load/adapt time. semver.Canonical can normalize valid semver; fall back to raw equality for non-semver tags. Pure-Go (golang.org/x/mod/semver already imported), cross-platform.

### parser_js/xmlutil-tojson-stack-overflow

- **Title:** xmlutil XML-to-JSON recursion has no depth bound — deeply nested XML triggers unrecoverable stack overflow (DoS)
- **Location:** `internal/cli/xmlutil/xmlutil.go:312`  ·  **CWE:** CWE-674 (Uncontrolled Recursion)  ·  **Verdict:** 3/3 lenses, reproduced=True  ·  **Fix invariant-safe:** True
- **What:** RunFromXML parses untrusted XML (stdin, file, or literal arg) via parseXML, which builds an arbitrarily deep *xmlNode tree iteratively (no depth limit on xml.Decoder.Token nesting), then calls (*xmlNode).toJSON, which recurses once per nesting level (line 341: childJSON := child.toJSON(opts)) with NO maximum-depth guard. Go's encoding/xml does NOT bound element nesting, so a document of N nested elements produces N stack frames in toJSON. This is the SAME bug class that HARDENING.md parsing-02 fixed for the yq YAML parser (maxYAMLDepth=1000), but xmlutil was left unguarded. I confirmed it empirically: depth 100000 survived (encoder hit its own cap), but at ~500000 nested elements the process died with 'runtime: goroutine stack exceeds 1000000000-byte limit / fatal error: stack overflow' inside xmlutil.toJSON at xmlutil.go:341. A stack overflow is a fatal Go runtime abort that recover() cannot catch and that bypasses cmderr exit-code mapping — the whole omni process crashes.
- **Exploit:** Build a few-MB file of deeply nested tags: printf into deep.xml roughly 500000 copies of '<a>' followed by 'x' followed by 500000 copies of '</a>', then run `omni xml fromxml deep.xml` (or `omni json fromxml` whatever the registered verb is). The toJSON recursion overflows the goroutine stack and the process aborts with 'fatal error: stack overflow'. Because getInput uses os.ReadFile for file args (and the multi-line form bypasses the bufio.Scanner per-line cap), the payload is trivial to deliver in CI when omni is used to convert an attacker-supplied XML artifact.
- **Fix:** Thread a depth counter through (*xmlNode).toJSON (and ideally cap tree depth during parseXML by tracking len(stack)) and return cmderr.ErrInvalidInput once a sane cap (e.g. 1000, matching maxYAMLDepth in yq) is exceeded — e.g. add a maxXMLDepth const and a toJSON(opts, depth int) signature that errors past the cap, propagating the error out of parseXML/RunFromXML. Pure-Go, stdlib-only, cross-platform; mirrors the existing yq fix exactly and adds no deps. Optionally also reject in parseXML when len(stack) exceeds the cap so the tree is never built that deep.

### parser_js/hls-segment-unbounded-readall

- **Title:** HLS downloadSegment reads response with bare io.ReadAll, bypassing the 128 MiB body cap (memory-exhaustion / gzip-bomb DoS)
- **Location:** `pkg/video/downloader/hls.go:192`  ·  **CWE:** CWE-409 (Improper Handling of Highly Compressed Data / Decompression Bomb)  ·  **Verdict:** 3/3 lenses, reproduced=True  ·  **Fix invariant-safe:** True
- **What:** The nethttp Client added readAllLimited (128 MiB cap, client.go:24-37) and GetString/GetJSON/PostJSON were migrated to it (this closed HARDENING.md netvideo-01). But HLSDownloader.downloadSegment issues its own request via opts.Client.Do(req) and then reads the body with a bare data, err := io.ReadAll(resp.Body) at hls.go:192 — it does NOT go through readAllLimited and has NO io.LimitReader or size cap. The segment URL comes verbatim from the (untrusted) M3U8 manifest (SSRF is guarded at line 168 by validateFetchURL, but size is not). The shared nethttp transport leaves transparent gzip decompression enabled, so a malicious/MITM'd manifest can point a segment at a server returning a small gzip body that net/http inflates to gigabytes, and the whole inflated segment is buffered into memory by io.ReadAll, exhausting RAM. This is a spot the netvideo-01 fix missed.
- **Exploit:** User (or CI) runs `omni video <hls-or-page-url>` against a hostile/compromised HLS source (or a MITM rewrites the manifest in transit). The attacker lists a normal-looking #EXTINF segment whose http(s) URL (passes validateFetchURL) returns Content-Encoding: gzip with a tiny payload that decompresses to multiple GB; downloadSegment's io.ReadAll(resp.Body) at hls.go:192 buffers the entire decompressed segment in memory, OOM-killing the omni process on the operator's machine.
- **Fix:** Route this read through the same bounded reader used elsewhere: replace io.ReadAll(resp.Body) with the package's readAllLimited (export it or add an exported Client.GetBytes that fetches+limits), or wrap with io.ReadAll(io.LimitReader(resp.Body, maxSegmentBytes+1)) and error when the cap is hit. Pick a per-segment ceiling (e.g. reuse maxResponseBytes = 128 MiB). Pure-Go, stdlib-only, cross-platform, no new deps; identical pattern to the existing nethttp fix.

### ssrf/scan-dbupdate-unbounded-readall-no-timeout

- **Title:** scan db update fetch uses default http.Get: unbounded io.ReadAll, no timeout, unguarded redirect-following before signature verification
- **Location:** `internal/cli/scan/scan.go:224`  ·  **CWE:** CWE-400: Uncontrolled Resource Consumption (with CWE-409 decompression/memory-bomb and CWE-918 SSRF facets)  ·  **Verdict:** 3/3 lenses, reproduced=True  ·  **Fix invariant-safe:** True
- **What:** RunDBUpdate downloads the OSV bundle via fetch(url), which calls http.Get(url) (the zero-Timeout http.DefaultClient) and then io.ReadAll(resp.Body) with no io.LimitReader and no request context/deadline. The full osv-db.zip and .minisig are read entirely into memory BEFORE scan.VerifyAndLoadDB runs, so the signature fail-closed guarantee does NOT protect this path: the server's response body is untrusted even when the --url host is trusted. Three concrete gaps: (1) no body-size cap — a hostile/compromised mirror or a MITM on the leg to the redirect target streams an arbitrarily large (or transparently gunzipped) body and exhausts memory; (2) no client/request timeout — a slow-loris server hangs the omni process indefinitely with no way to cancel; (3) http.DefaultClient follows up to 10 redirects with zero scheme/host validation. Redirect-following is load-bearing here (the default --url https://github.com/inovacc/omni/releases/latest/download serves a 302 to objects.githubusercontent.com), so a malicious --url or a MITM/compromised redirect can bounce the fetch into an internal/link-local/metadata endpoint (169.254.169.254) — an SSRF probe that fires before any signature check. This is notably weaker than the sibling video client (pkg/video/nethttp/client.go), which the prior sweep hardened with readAllLimited (128 MiB cap) AND a 30s timeout AND validateFetchURL/isPublicIP SSRF guards; the new scan fetch path got none of those.
- **Exploit:** Operator runs `omni scan db update --url https://attacker-mirror.example/osv --db-key trusted.pub` (or a trusted --url whose TLS leg is MITM'd, or whose 302 redirect target is attacker-controlled). The server returns 200 and streams a multi-gigabyte body (or a small gzip that net/http transparently inflates); io.ReadAll buffers it all with no cap and no timeout, OOM-ing or hanging the omni process before VerifyAndLoadDB is ever reached. Variant: the server replies 302 Location: http://169.254.169.254/latest/meta-data/iam/security-credentials/ and omni follows it unvalidated, issuing the internal request (blind SSRF probe / hang) prior to signature verification.
- **Fix:** Replace the bare http.Get with an http.Client that has an explicit Timeout (e.g. 60s) and pass a cancellable context via http.NewRequestWithContext. Wrap resp.Body in io.LimitReader with a sane ceiling (e.g. 256 MiB, generously above any real OSV bundle) and treat hitting the cap as cmderr.ErrIO. Add a CheckRedirect that caps hop count and rejects redirect targets whose host is a literal private/loopback/link-local IP or resolves only to such ranges — reuse the existing isPublicIP/validateFetchURL logic from pkg/video/downloader/hls.go (or factor it into a shared pure-Go helper) so 169.254.169.254 and RFC1918/loopback are refused. All pure-Go/stdlib (net, net/http, net/url, io), cross-platform, no new deps, no exec; preserves the fail-closed-then-verify ordering.

### intoverflow/ws-frame-neg-payloadlen-panic

- **Title:** WebSocket frame parser decodes 64-bit length into signed int, negative value bypasses bounds guard and panics on slice
- **Location:** `internal/cli/video/auth.go:408`  ·  **CWE:** CWE-190: Integer Overflow or Wraparound (leading to CWE-129 improper validation of array index / out-of-bounds slice panic)  ·  **Verdict:** 2/3 lenses, reproduced=True  ·  **Fix invariant-safe:** True
- **What:** wsExtractPayload (auth.go:387-432) parses a WebSocket frame from a Chrome DevTools Protocol connection. For the 127 extended-length case it builds payloadLen with a hand-rolled big-endian shift loop `payloadLen = payloadLen<<8 | int(data[2+i])` over 8 bytes into a Go `int`. On 64-bit platforms `int` is signed 64-bit, so any 8-byte length with the high bit set (e.g. 0xFFFFFFFFFFFFFFF6) yields a NEGATIVE payloadLen. The subsequent guard `if len(data) < offset+payloadLen { return nil }` does not catch this: with a negative payloadLen, `offset+payloadLen` is smaller than offset (and <= len(data)), so the guard passes. The code then executes `payload := data[offset : offset+payloadLen]`, where the high index is less than the low index, triggering a runtime panic (slice bounds out of range [offset:offset+payloadLen]). There is no recover() anywhere in the auth call chain nor at top-level dispatch (coreio-01 panic-recovery is still open per HARDENING), so the entire omni process aborts with a Go runtime stack trace and exit code 2. The frame bytes flow directly from `conn.Read(readBuf)` (auth.go:318) reading the CDP WebSocket response into `responseData`, which is passed verbatim to wsExtractPayload (auth.go:322). gosec G115 does NOT flag this because the overflow is in a manual shift loop rather than a single int(...) conversion cast.
- **Exploit:** Run the YouTube cookie-auth path (omni video auth / cookie extraction) which connects to a Chrome DevTools Protocol WebSocket. A hostile or compromised CDP endpoint, a man-in-the-middle on the loopback WS, or simply a malformed CDP server response sends a single frame: byte0=0x01 (opcode), byte1=0x7F (len marker 127, unmasked), then 8 length bytes 0xFFFFFFFFFFFFFFF6 (= -10) — i.e. `{0x01,0x7F,0xFF,0xFF,0xFF,0xFF,0xFF,0xFF,0xFF,0xF6}`. payloadLen becomes -10, offset=10, offset+payloadLen=0, the bounds guard `len(data)<0` is false, and `data[10:0]` panics with `slice bounds out of range [10:0]`, crashing the omni process. Reproduced locally for both unmasked and masked (byte1=0xFF) variants.
- **Fix:** After decoding payloadLen in both the 126 and 127 branches, validate it is non-negative and within a sane ceiling before any slicing or allocation, e.g.: `if payloadLen < 0 || payloadLen > maxWSFrame { return nil }` (maxWSFrame e.g. 16<<20). For the 127 branch, since a CDP control/data frame will never legitimately exceed a few MB, additionally reject lengths above the ceiling. Also re-check `offset+payloadLen` for overflow using the now-validated non-negative payloadLen (guard `offset > len(data)-payloadLen`). Pure-Go, stdlib-only, cross-platform (the bug and fix are identical on Linux/macOS/Windows and on amd64/arm64 since Go int is 64-bit on all 64-bit targets). No new deps. This is a self-contained input-validation fix in wsExtractPayload.

### toctou/cp-no-mode-preserve
